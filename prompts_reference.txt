PROMPT 0: INITIAL COMPLETE BASE APPLICATION SPECIFICATION
========================================================================
Below is the comprehensive initial prompt used to design and build the foundation of this Weather Application, centering heavily on accessibility, interactive meteorology, and compliance reporting.

● THE ORIGINAL COMPREHENSIVE SPECIFICATION:
"Build a highly advanced, ultra-accessible (WCAG 2.1 AA/AAA compliant) Weather Dashboard application in React, Tailwind, and TypeScript. The application must not only act as an elegant, fluid weather tracker but serve as a masterclass in modern web accessibility.

Key Functional Modules & Constraints to Include:
1. Real-Time Meteorology Integration:
   - Connect directly to the Open-Meteo API to retrieve actual current, hourly, and 7-day meteorological data based on user queries.
   - Support city search, geolocation requests, and standard error fallback states.
   - Include persistent storage (localStorage) to cache searched locations and user preferences for offline-ready telemetry.

2. Severe Weather Alert Hub:
   - Design an active warning panel that alerts users to severe meteorological conditions (e.g., severe typhoons, extreme UV index heatwaves, frost storms).
   - Play high-readability visual alert banners and implement non-intrusive sound pings/notifications.

3. Automated A11y Audit Suite:
   - Build a real-time, interactive compliance scanner directly inside the application!
   - Users must be able to run a 'Compliance Audit' that dynamically tests the current DOM structure, landmarks, ARIA labels, focus states, and color contrast.
   - Display a beautiful radial scoreboard (0-100%) indicating the application's compliance rate, complete with a detailed diagnostic checklist showing passed/failed criteria and recommended remediations.

4. Developer Documentation Portal:
   - Provide clean, searchable interactive documentation inside the app covering:
     - WAI-ARIA Landmark roles used in the layout.
     - Semantic HTML element best practices.
     - Screen Reader live announcements.
     - Accessibility testing methods.

5. Accessibility & Assistant Tooling:
   - Built-in Voice Announcer Engine: Provide full text-to-speech support using the Web Speech Synthesis API. The app must dynamically dictate active weather updates, alerts, search completions, and widget configurations to users.
   - Design a 'High Contrast Mode' toggle that instantly swaps out standard soft design tokens with extremely high-contrast borders, solid black/white/yellow elements, and heavy-weight typography.
   - Keep interactive areas accessible with a minimum of 44x44px touch targets and distinct focus-ring styling ('accessible-focus').

6. Customizable Forecast Widget Engine:
   - Allow users to custom-configure their hourly forecast display via a dedicated panel:
     - Swappable layout modes: Responsive Grid, Horizontal Scrolling Carousel, or Compact Minimal List view.
     - Active metric filters: Checkbox filters enabling users to toggle visibility of temperature, apparent temperature, humidity, and wind speed.
     - Display duration constraints: Toggle outlook limits (e.g., 6h, 12h, or 24h outlook).

7. 7-Day Temperature Trend Visualizer:
   - Render an interactive, beautifully detailed graph showing temperature highs, lows, and precipitation envelopes using Recharts.
   - Ensure the chart is accompanied by a readable semantic table or complete screen reader narration to maintain WCAG-level data accessibility."

PROMPT 1: Dynamic Recharts Temperature Color Refactoring
------------------------------------------------------------------------
● USER REQUEST:
"Refactor the Recharts component to dynamically update its color palette (e.g., using cooler tones for below-freezing and warmer tones for heatwaves) to match the current theme-based app colors."

● ANALYSIS & PLANNING:
The temperature chart previously used static orange (High) and blue (Low) stroke/gradient definitions. To make it dynamic, we need to read the current temperature and weather conditions, then return customized stroke, fill gradient stop colors, grid line opacities, and legend/badge tailwind classes.

● IMPLEMENTATION COMPLETED:
1. Created `getChartColors()` helper in `WeatherDashboard.tsx`:
   - Freezing / Snow Tones (<= 0°C or Snow): Frost blue and crystal cyan gradients.
   - Heatwaves (>= 32°C): Intense warm crimson and deep orange gradients.
   - Storms: Moody purple and violet gradients.
   - Rain: Heavy indigo and deep blue gradients.
   - Cloudy: Sleek slate and steel grey gradients.
   - Default Warm (>= 18°C) / Cool (< 18°C): Custom warm amber/orange vs. cool sky/breeze blue gradients.
2. Refactored the `Recharts` code:
   - Configured Dynamic linearGradients (`#colorHigh` and `#colorLow`) using computed stop colors.
   - Swapped out hardcoded string colors on `<Area>`, `<XAxis>`, `<YAxis>`, `<CartesianGrid>` with active chart palette variables.
   - Synchronized HTML tooltips, legend dots, and badges using dynamic Tailwind class injections.

------------------------------------------------------------------------
PROMPT 2: Expandable Current Conditions Card ('Show More')
------------------------------------------------------------------------
● USER REQUEST:
"Add a 'Show More' toggle button in the current conditions card that expands to reveal atmospheric pressure and visibility metrics, keeping the initial view clean and accessible."

● ANALYSIS & PLANNING:
The dashboard should present a tidy, uncluttered primary view. Less critical weather details like UV Index, Rain Chance, Pressure, and Visibility should start collapsed and expand gracefully on demand. We also need to acquire actual 'visibility' telemetry from Open-Meteo API since it wasn't fetched initially.

● IMPLEMENTATION COMPLETED:
1. Adjusted `src/types.ts`:
   - Updated `CurrentWeather` interface to support an optional `visibility?: number` field.
2. Updated `src/App.tsx`:
   - Modified Open-Meteo query to fetch `current=...,pressure_msl,visibility`.
   - Handled real-time telemetry extraction, defaulting visibility safely to 10,000 meters if absent.
3. Expanded `WeatherDashboard.tsx` with dynamic controls:
   - Added `showMoreDetails` React state boolean.
   - Imported Lucide Icons `ChevronDown` and `ChevronUp`.
   - Designed a polished expander button utilizing dynamic CSS and appropriate ARIA attributes for accessibility (`aria-expanded`, `aria-controls`).
   - Integrated Framer Motion `<AnimatePresence>` and `<motion.div>` for a smooth sliding height/opacity transition.
   - Included reactive Screen Reader status announcements via `announceSpeech()` helper upon expand/collapse actions.

------------------------------------------------------------------------
PROMPT 3: WCAG Contrast AA/AAA Readiness
------------------------------------------------------------------------
● USER REQUEST:
"WCAG Contrast AA/AAA Readiness"

● ANALYSIS & PLANNING:
Several UI elements previously relied on low-contrast `slate-400` or `slate-500` colors on light/dark backgrounds (failing WCAG minimum 4.5:1 ratio for normal text). Under High Contrast Mode, these need to be forced to pure black/white, high-intensity yellow accents, or high-weight bold text.

● IMPLEMENTATION COMPLETED:
1. Conducted an audit of muted text in `WeatherDashboard.tsx`.
2. Replaced low-contrast subheaders, label indicators, and caption strings with dynamic conditional checks:
   - Location Lat/Lon: Shifted from slate-500 to bold white or bright yellow under High Contrast Mode.
   - Apparent Temp and Fetch indicators: Overrode muted gray with bright contrast colors.
   - Chart Description & Guide captions: Replaced static slate-400 text with high-contrast text styles.
   - Customizable Hourly Widget & Forecast list items: Upgraded contrast for layouts, hourly labels, wind metrics, and low-temp ranges.
   - Dashboard Favorites Bar: Improved label contrast for easy visibility.
